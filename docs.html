Introduction

Welcome to CookiEzu

The lightweight, GDPR-compliant cookie consent plugin for WordPress. Open source, free forever, and beautiful by default. Now at v1.1.0 with all layout types fully working.

CookiEzu gives your WordPress site a fully compliant cookie consent experience. When a visitor lands on your site, a customisable banner appears asking for their consent before any non-essential cookies are loaded. Their choice is remembered, logged, and respected on every future visit.

‚úÖ
GDPR, CCPA & PDPA ReadyCookiEzu is built to satisfy the consent requirements of major privacy laws including EU GDPR, California CCPA, and Southeast Asian PDPA frameworks.

Installation

Installing CookiEzu

Get the plugin installed on your WordPress site in under 2 minutes.

üÜï
Download v1.1.0 ‚Äî This version fixes the modal dismiss bug and all layout types now work correctly. If you have an older version installed, follow the upgrade steps below.

Method 1 ‚Äî Upload via WordPress Admin

Download the plugin ZIP
Go to github.com/flyzal/CookiEzu, click Releases, and download the latest cookiezu.zip file.
Go to Plugins ‚Üí Add New
In your WordPress admin dashboard, navigate to Plugins ‚Üí Add New Plugin.
Upload the ZIP
Click Upload Plugin at the top, choose your cookiezu.zip file, and click Install Now.
Activate
After installation completes, click Activate Plugin. CookiEzu will create its database table automatically.
You're done!
A new CookiEzu menu item will appear in your WordPress sidebar. Click it to configure your banner.

Upgrading from v1.0.0

‚ö†Ô∏è
Important upgrade steps ‚Äî Simply re-uploading over an existing install may leave cached old JS files. Follow these steps exactly to guarantee the fix takes effect.
Deactivate the old plugin
Go to Plugins ‚Üí Installed Plugins, find CookiEzu, and click Deactivate.
Delete the old plugin
Click Delete on the CookiEzu row. Your settings and consent log are stored in the database and will NOT be deleted.
Upload and activate v1.1.0
Follow Method 1 above to install the fresh v1.1.0 ZIP. All your previous settings will be intact.
Verify in an incognito window
Open your site in a private/incognito browser window. The banner should appear and dismiss correctly when you click any button.

Method 2 ‚Äî FTP / Manual

Extract the ZIP

Upload via FTP
Using FileZilla or your hosting file manager, upload the cookiezu/ folder to /wp-content/plugins/.
Activate in WordPress
Go to Plugins ‚Üí Installed Plugins, find CookiEzu, and click Activate.
Docs‚Ä∫Quick Setup
Quick Setup

Configure your banner

A 5-minute walkthrough of the most important settings to get your banner live and compliant.

Step 1 ‚Äî Set your banner text

Go to CookiEzu ‚Üí Settings ‚Üí Content tab. Update the Banner Title and Banner Message to reflect your site's brand. Be clear and honest about what cookies you use.

üí°
Good message example: "We use cookies to improve your experience, analyse traffic, and serve personalised content. You can choose which categories to accept."

Step 2 ‚Äî Add your Privacy Policy link

Still on the Content tab, paste your Privacy Policy page URL into the Privacy Policy URL field. This is required under GDPR ‚Äî visitors must be able to read your full policy from the banner.

Step 3 ‚Äî Choose your layout

Go to the Design tab and pick a Layout and Position. For most sites, Bar + Bottom is the least intrusive. Modal is best if you need maximum visibility for strict compliance. All three layouts now dismiss correctly in v1.1.0.

Step 4 ‚Äî Enable the categories you use

Go to the Categories tab. Only enable the categories that match cookies you actually load. If you don't use Facebook Pixel, don't show a Marketing toggle ‚Äî it will confuse visitors.

Step 5 ‚Äî Enable consent recording

Go to the Advanced tab and make sure Record Consent is ticked. This builds your GDPR audit trail in CookiEzu ‚Üí Consent Log.

Step 6 ‚Äî Save and test

Click Save Settings. Then open your website in a private/incognito browser window to see the banner as a first-time visitor would. This is important ‚Äî your own browser likely has a consent cookie already stored.

Docs‚Ä∫What's New
Changelog

v1.1.0 ‚Äî Modal Fix

Released February 2026. Critical bug fixes ensuring all three layout types (Bar, Box, Modal) dismiss correctly when a visitor makes a consent choice.

üêõ
What was broken in v1.0.0 ‚Äî The modal layout's display: flex !important CSS rule was overriding the JavaScript's attempts to hide the banner via a CSS class toggle. The result: clicking any button appeared to do nothing visually.

Changes in this release

Modal consent dismiss fixed Bug Fix
The modal no longer stays open after clicking Accept All, Necessary Only, or Save Preferences. Root cause: CSS display: flex !important on .cookiezu-layout-modal was winning the specificity battle against the JS class-based hide system. Fixed by switching JS to use element.style.display (inline styles always beat CSS rules).
Preferences panel now opens correctly Bug Fix
Clicking "Customize" now reliably shows the cookie category toggles panel. The panel was blocked by a hardcoded style="display:none" in the PHP template that the JS's class-removal approach could never override.
Rewritten JS show/hide engine Improvement
Replaced the fragile CSS class toggle system (cz-hidden) with direct element.style.display manipulation. Inline styles set by JavaScript always take precedence over any stylesheet rule regardless of specificity or !important ‚Äî this is guaranteed by the CSS cascade specification.
Correct display value per layout type Improvement
The new bannerDisplayValue() function detects whether the banner uses the modal layout (needs display: flex for centering) vs bar/box layouts (needs display: block). Previously all layouts used the same value, breaking the modal's centred positioning.
Removed conflicting inline styles from banner.php Bug Fix
All style="display:none" attributes removed from the PHP banner template. Visibility is now 100% controlled by JavaScript, removing the root cause of the CSS vs inline style conflict.
Version bump to 1.1.0 for cache busting Improvement
Plugin version number incremented in both the main plugin file header and the COOKIEZU_VERSION constant. WordPress uses this version as a query string on enqueued assets (?ver=1.1.0), forcing browsers and CDNs to load the new JS and CSS files even if the old ones were cached.

Consent Log

Your GDPR audit trail. Every consent decision recorded, stored, and ready if you're ever asked to prove compliance.

When Record Consent is enabled in the Advanced settings, CookiEzu logs every consent event to a dedicated database table (wp_cookiezu_consent_log).

What Gets Recorded

FieldDescription
idUnique record ID
consent_dateExact date and time of the consent event (server time)
ip_addressVisitor IP address ‚Äî last 4 characters masked for privacy (e.g. 192.168.1.****)
necessaryAlways 1 (true)
analytics1 = accepted, 0 = declined
marketing1 = accepted, 0 = declined
functional1 = accepted, 0 = declined
‚ÑπÔ∏è
Where to find it: Go to WordPress Admin ‚Üí CookiEzu ‚Üí Consent Log. The table shows the latest 200 records. The header shows your total record count.

Consent Expiry

The Consent Expiry setting (in Advanced) controls how long the visitor's cookie lasts in their browser. Default is 365 days. After expiry, the banner re-appears and a new consent record is created. GDPR recommends re-obtaining consent at least annually.

Docs‚Ä∫Integrations‚Ä∫Google Analytics 4
Integration

Google Analytics 4

Auto-load GA4 with full Consent Mode v2 support ‚Äî no additional plugins or code needed.

How It Works

When you enter a GA4 Measurement ID, CookiEzu automatically injects the gtag.js script and sets up Consent Mode v2 ‚Äî Google's official privacy framework. By default, all consent signals are set to denied. When a visitor accepts analytics cookies, CookiEzu updates the signals to granted and GA4 begins full tracking.

‚úÖ
This keeps you compliant ‚Äî GA4 won't set any cookies or track behaviour until the visitor explicitly consents to Analytics cookies.

Setup

Create a GA4 property
Go to analytics.google.com ‚Üí Admin ‚Üí Create Property. Choose Google Analytics 4.
Copy your Measurement ID
In your GA4 property, go to Admin ‚Üí Data Streams ‚Üí your stream. Copy the Measurement ID (format: G-XXXXXXXXXX).
Paste into CookiEzu
Go to CookiEzu ‚Üí Settings ‚Üí Integrations. Paste your ID into the Google Analytics 4 ID field and save.
Verify
Visit your site, accept analytics cookies, then check the Realtime report in GA4 ‚Äî you should see yourself as an active user.
‚ö†Ô∏è
Don't use another GA4 plugin at the same time. If you already load GA4 via MonsterInsights or Site Kit, remove that first ‚Äî otherwise GA4 will load twice and data will be doubled.

Google Tag Manager

CookiEzu pushes consent events to GTM's dataLayer, letting you control any tag based on visitor consent ‚Äî without touching code.

Setup

Get your GTM Container ID
Go to tagmanager.google.com. Your container ID is in the top right (format: GTM-XXXXXXX).
Enter it in CookiEzu
Go to CookiEzu ‚Üí Settings ‚Üí Integrations ‚Üí Google Tag Manager ID and save.
Create triggers in GTM
In GTM, create a Custom Event trigger that fires on cookiezu_consent_updated. Use it to fire tags like Facebook Pixel or Hotjar only when the relevant consent category is granted.

dataLayer Event Structure

JavaScript
{
  event: "cookiezu_consent_updated",
  cookiezu: {
    necessary:  true,
    analytics:  true,   // or false
    marketing:  false,
    functional: true,
    version:    "1.1.0",
    date:       "2025-03-01T10:22:00.000Z"
  }
}

In GTM you can access these values as {{dlv - cookiezu.analytics}} etc. to conditionally fire tags.

Integration

JavaScript API

Load any third-party script conditionally based on what the visitor consented to ‚Äî no GTM required.

The Consent Event

Every time a visitor saves their consent preferences, CookiEzu fires a custom DOM event called cookiezuConsentUpdated. You can listen for this anywhere in your theme or plugin code.

JavaScript
document.addEventListener('cookiezuConsentUpdated', function(e) {
  const consent = e.detail;

  // consent.necessary  ‚Üí always true
  // consent.analytics  ‚Üí true or false
  // consent.marketing  ‚Üí true or false
  // consent.functional ‚Üí true or false

  if (consent.analytics) {
    loadHotjar(); // Load Hotjar, Plausible, etc.
  }
  if (consent.marketing) {
    loadFBPixel(); // Load Facebook Pixel
  }
  if (consent.functional) {
    loadIntercom(); // Load Intercom chat widget
  }
});

Reading Consent Without the Event

You can also read the current consent state directly from the browser cookie:

JavaScript
function getCookieZuConsent() {
  const match = document.cookie.match(/cookiezu_consent=([^;]*)/);
  if (!match) return null;
  try { return JSON.parse(decodeURIComponent(match[1])); }
  catch(e) { return null; }
}

const consent = getCookieZuConsent();
if (consent?.analytics) {
  // analytics already accepted ‚Äî load scripts
}

Docs‚Ä∫Advanced‚Ä∫Troubleshooting
Advanced

Troubleshooting

Diagnosing and fixing the most common issues with CookiEzu.

Modal doesn't close when clicking buttons

üêõ
This was a known bug in v1.0.0. Upgrade to v1.1.0 ‚Äî the fix is included. See the Changelog for full details.

Banner doesn't appear at all

The most common cause is that you (or a visitor) already have a consent cookie stored. Open your browser DevTools ‚Üí Application tab ‚Üí Cookies and look for cookiezu_consent. Delete it and refresh. Always test in a private/incognito window to simulate a first-time visitor.

Banner appears but JS seems broken (buttons unresponsive)

Open DevTools ‚Üí Console tab and look for red JavaScript errors. Common causes: a caching plugin serving the old JS file, a theme or other plugin causing a JS conflict, or the plugin files being corrupted during upload. Try deactivating other plugins one-by-one to isolate a conflict. Also try clearing your caching plugin's cache.

After upgrading, the old behaviour persists

A caching plugin (WP Super Cache, W3 Total Cache, LiteSpeed Cache, etc.) may be serving the old JS file. Clear all caches after upgrading. Also check if your hosting has server-level caching (common on managed WordPress hosts like Kinsta, WP Engine, Flywheel) and clear that too.

üí°
Quick diagnostic: In DevTools Console, run fetch('/wp-content/plugins/cookiezu/public/js/cookiezu-public.js').then(r=>r.text()).then(t=>console.log(t.slice(0,200))). If the output contains cz-hidden you're still on the old version. If it contains style.display you have v1.1.0.

Consent log is empty

Make sure Record Consent is enabled in CookiEzu ‚Üí Settings ‚Üí Advanced. Also check that your WordPress AJAX URL is correct ‚Äî test by running console.log(cookiezuSettings.ajaxUrl) in the browser console. If it's empty or wrong, there may be a conflict with your permalink structure.

GA4 isn't tracking after consent

First, check you haven't got another GA4 plugin loaded (MonsterInsights, Site Kit, etc.) ‚Äî they conflict. Then open DevTools ‚Üí Network tab, accept analytics cookies, and look for a request to google-analytics.com or googletagmanager.com. If there's no network request, the GA ID may be incorrectly entered. Check it matches the G-XXXXXXXXXX format exactly.

Docs‚Ä∫FAQ
FAQ

Frequently Asked Questions

Quick answers to the most common questions about CookiEzu.

Will CookiEzu slow my site down?

No. The front-end JavaScript is under 4KB with zero dependencies. The CSS is ~6KB. Both load asynchronously in the footer and have no impact on your Core Web Vitals or page speed score.

Can I use CookiEzu with WooCommerce?

Yes. WooCommerce's session cookies are Necessary cookies and don't require consent. You only need to show the Marketing category if you run WooCommerce Ads or remarketing. The plugin works seamlessly alongside WooCommerce.

Why does the banner show again after I dismissed it?

This happens when you're in a private/incognito window (cookies are wiped when you close it), or when the consent cookie has expired. Check your Consent Expiry setting in the Advanced tab.

Does CookiEzu block scripts automatically?

In v1.1.0, CookiEzu fires the cookiezuConsentUpdated JavaScript event which you use to load scripts conditionally. Automatic script blocking (changing type="text/javascript" to type="text/plain") is planned for v2.

Is this plugin free forever?

Yes. CookiEzu is released under the GPL v2 license. You can use it on unlimited sites, modify it, and redistribute it. There are no paid tiers, no usage limits, and no subscriptions.

How do I report a bug or request a feature?

Open an issue on github.com/flyzal/CookiEzu/issues. Pull requests are also very welcome!

Why do I need to delete the plugin before upgrading?

Some caching plugins and WordPress installations keep the old files on disk even after a re-upload. Deleting first guarantees a clean slate. Your settings and consent log are stored in the WordPress database and are never deleted when you remove the plugin files.

Contributing

CookiEzu is built by the community, for the community. Every bug report, pull request, translation, and star makes it better for everyone.

How Contributions Work

CookiEzu uses the standard open source fork-and-pull model ‚Äî the same workflow used by WordPress, VS Code, and virtually every major open source project. You do not need a direct invitation to contribute. Anyone can fork, improve, and submit a pull request.

‚ÑπÔ∏è
On "GitHub Collaborators": Being added as a direct repository collaborator means having push access to the main repo. This is reserved for long-term trusted maintainers who have already contributed significantly through PRs. The fork-and-pull workflow is the correct path for new contributors ‚Äî it is safer for the project and still gives you full credit for your work.

The Fork-and-Pull Workflow

Fork the repository
Go to github.com/flyzal/CookiEzu and click Fork in the top-right corner. This creates a personal copy of the repo under your own GitHub account.
Create a descriptive branch
Clone your fork locally. Create a branch named for your change: git checkout -b fix/modal-dismiss or git checkout -b feature/auto-scanner. Never commit directly to main.
Make your change
Write clean, well-commented code following WordPress PHP coding standards. Test your change on a real WordPress install ‚Äî ideally across multiple themes and PHP versions.
Open a Pull Request
Push your branch to your fork, then open a Pull Request against flyzal/CookiEzu:main. Write a clear title and description. Link to any related Issues. The maintainer will review, may request changes, and merge when ready. GitHub automatically shows a link to the CONTRIBUTING guidelines when you open any Issue or PR.
